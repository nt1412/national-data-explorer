One engine, not three products

Every dataset here is a thin adapter over shared infrastructure: a census-code entity spine, a DuckDB/Parquet build, a unit-aware & polarity-adjusted change function, and static client-side maps. A new dataset is a new adapter; a new country is a new boundary set + code spine. Nothing above is India-specific by design — the roadmap is to map the world.

new dataset → datasets/<name>/{adapter, indicators}
new country  → boundaries + code spine → reuse everything else
  • Join on codes, never names (spelling variants silently drop rows)
  • New/uncovered units shown as “no baseline”, never imputed
  • Change is polarity- and unit-aware (points vs. rates vs. ratios)
  • Colourblind-safe diverging & sequential ramps; dark mode
  • Source & licence recorded per layer (OSM/ODbL kept isolable)
Honest scope. These are real government datasets with real limits, documented in each explorer: NFHS-5 covers ~641 comparable districts; road-safety severity is in-hand but per-capita/per-vehicle rates are a fast-follow; the infrastructure overlay is Maharashtra-only and OSM-completeness-biased. A national district industry layer is sourced but blocked at the national level (state Economic-Census data is gated) — deferred, not faked.

Data: IIPS/MoHFW (NFHS) · MoRTH / Transport Research Wing (road safety) · OpenStreetMap contributors, ODbL (infrastructure) · OurAirports (public domain) · DataMeet, CC-BY (boundaries). Attribution shown in-context on each map.

Build: static, client-side, zero-backend — deployable to any static host. See the repository README and per-dataset specs for sourcing, the verified data corrections, and how to add a country or dataset.
